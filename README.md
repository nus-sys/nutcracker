# NutCracker
Compilation Framework For Hybrid DPU

## Prerequisites 

Before building the project you should initialize and update the repository submodules (this will fetch the `third_party` sources used below):

```bash
git submodule update --init
```

### System Requirements

- **Rust toolchain** (version 1.74.0 or later)
  
  Install Rust using rustup:
  ```bash
  curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
  source $HOME/.cargo/env
  ```
  
  Verify installation:
  ```bash
  rustc --version  # Should show 1.74.0 or later
  cargo --version
  ```

- **CMake** (version 3.20 or later)
- **Ninja** build system

### Third-party Dependencies

To set up the required third-party dependencies, run the following commands:

1. **Initialize LLVM**
   ```bash
   bash build_scripts/build_llvm.sh
   export LLVM="$PWD/third_party/llvm/build/bin"
   ```

2. **Initialize P4MLIR**
    ```bash
    pushd third_party/p4mlir/build_tools/
    git apply ../../../patches/01_fix_p4mlir_build_script.patch
    bash build_p4c_with_p4mlir_ext.sh
    popd
    ```

3. **Initialize egglog**
    ```bash
   pushd third_party/egglog
   cargo build --release
   popd
   export EGGLOG="$PWD/third_party/egglog/target/release"
   ```

4. **Set environment variable:**
   ```bash
   export PATH=$LLVM:$EGGLOG:$PATH
   ```

## Building NutCracker

After setting up the third-party dependencies, build NutCracker:

```bash
mkdir -p build
cd build
cmake ..
make -j$(nproc)
```

## Pre-compilation Steps

Before using NutCracker, you need to compile your source code to the appropriate intermediate representation:

### Compiling P4 Programs to P4MLIR

For P4 programs, use the P4MLIR compiler to generate MLIR representation:

```bash
./third_party/p4mlir/third_party/p4c/build/p4mlir-translate \
  -I./p4include \
  --typeinference-only \
  <input.p4> > <output.mlir>
```

**Example:**
```bash
./third_party/p4mlir/third_party/p4c/build/p4mlir-translate \
  -I./p4include \
  --typeinference-only \
  apps/simple_fowarding/simple_forwarding.p4 > apps/simple_fowarding/simple_forwarding.mlir
```

### Compiling C/C++ Programs to LLVM IR

For C/C++ programs, use Clang to generate LLVM IR:

```bash
clang -S -emit-llvm -O2 <input.c> -o <output.ll>
```

**Example:**
```bash
clang -S -emit-llvm -O2 example.c -o example.ll
```

Alternatively, to generate MLIR from C/C++:

```bash
mlir-translate --import-llvm <input.ll> -o <output.mlir>
```

## Compiling a Sample App (end-to-end)

The recommended way to compile a sample app is via the `nutcracker.sh` script,
which drives the full P4 → P4HIR → vDRMT → bf3drmt → DOCA Flow pipeline:

```bash
bash nutcracker.sh compile <app_name>
```

**Example:**
```bash
bash nutcracker.sh compile shared_counter
```

This produces `vdsa_output/` with per-block MLIR artifacts and the generated
DOCA Flow C file `vdsa_output/doca_flow_pipeline.c`.

Available sample apps: `register`, `syn_flood`, `meter`, `load_balance`,
`simple_forwarding`, `shared_counter`.

## Building the Generated Pipeline (deploy/)

After `nutcracker.sh compile` succeeds, build the deployable runtime binary
from the `deploy/` directory (located at the repo root).

**Prerequisites (BlueField-3 BSP):** DOCA ≥ 2.8, DPDK, FlexIO, libibverbs,
libmlx5 (all available in the BlueField BSP / DOCA SDK).

```bash
cd deploy
meson setup build_deploy
meson compile -C build_deploy
```

This produces `deploy/build_deploy/nc_runtime` — an aarch64 ELF that links
against DOCA Flow, DPDK, and FlexIO.

> **Note:** `deploy/meson.build` references `../vdsa_output/doca_flow_pipeline.c`
> (generated by the last `nutcracker.sh compile` run) and the static runtime
> sources in `../runtime/`.  Re-run `nutcracker.sh compile` before rebuilding
> the deploy if you switch apps.

## Running the Pipeline on BlueField-3

DOCA does **not** support running DPA and DOCA Flow simultaneously.  Choose the
invocation based on which hardware the generated pipeline uses.

### All-DRMT pipeline (DOCA Flow HWS)

Used when all blocks are mapped to the NIC match-action engine (e.g.
`shared_counter`, `simple_forwarding`, `meter`).

```bash
sudo ./deploy/build_deploy/nc_runtime \
    -l 0-15 -n 4 \
    -a 03:00.0,dv_flow_en=2 \
    -a 03:00.1,dv_flow_en=2
```

`dv_flow_en=2` enables DPDK hardware-steering mode, required for DOCA Flow HWS.

### DPA-only pipeline

Used when blocks are mapped to the DPA accelerator.

```bash
sudo ./deploy/build_deploy/nc_runtime -l 0 -n 4 -a 03:00.0
```

### Expected output (DOCA Flow path)

A successful run prints messages similar to:

```
[NC_PIPELINE][INF] Created pipe 0 …
[NC_PIPELINE][INF] Created pipe 1 …
[NC_PIPELINE][INF] Created pipe 2 …
[NC_PIPELINE][INF] Created pipe 3 …
[NC_RUNTIME][INF] nutcracker runtime running (Ctrl-C to stop)
```

Press **Ctrl-C** to shut down gracefully.
